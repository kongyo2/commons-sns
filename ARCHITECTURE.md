# Cloudflare Workers構成

Commonsは、公式版のアプリケーション基盤をCloudflare上で完結させます。Google Cloudや常駐VPSは前提にしません。

| 役割 | Cloudflare製品 | 初期用途 |
| --- | --- | --- |
| Web、SSR、API | Workers | React Router、認証、投稿API、管理API |
| リレーショナルデータ | D1 | ユーザー、投稿、フォロー、リアクション、セッション |
| 画像・動画 | R2 | アバター、投稿メディア、OG画像 |
| 非同期処理 | Queues | 通知配信、メディア検査、集計更新 |
| エッジキャッシュ | Cache API | 公開プロフィール、公開投稿、OGレスポンス |
| 強い直列化が必要な状態 | Durable Objects | レート制御、リアルタイム通知など必要になった時点で導入 |

## リクエストの流れ

1. Workersが静的アセットまたはReact RouterのSSRを返す。
2. loader/actionからBindings経由でD1とR2へ直接アクセスする。
3. 投稿作成などの副作用はQueuesにイベントを送り、通知や集計をリクエスト外で処理する。
4. 公開データだけをCache APIへ載せ、ログイン状態のレスポンスはキャッシュしない。

## 初期の境界

- D1を唯一の正本とし、KVをユーザー状態の正本にはしない。
- R2へは原則として署名付きアップロードを使い、Workerを大容量ファイルの中継にしない。
- タイムラインは最初はD1の時系列クエリで構築する。
- 規模が増えた段階で、候補生成・ファンアウト・検索を個別に分離する。
- 外部コントリビューターがCloudflareアカウントなしでも開発できるよう、Wranglerのローカルエミュレーションを標準にする。

## 認証方針（未確定）

Cloudflareだけで任意の確認メールを送る機能は限定的なため、完全Cloudflare構成を優先する場合はPasskeyとリカバリーコードを第一候補にします。メール認証を必須にする場合は、送信サービスだけを外部依存として明示します。
